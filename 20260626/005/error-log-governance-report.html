🔔 告警规则建议

告警项级别筛选条件
连接 EOF + stream no originImageData WARN logger ~ "c\\.f\\.f\\..*" AND (message CONTAINS "connection EOF" OR "request body read EOF" OR "expected error" OR "demoted from com.github.trace")
Read timed out ERROR message CONTAINS "Read timed out"
远端 Stone 500 ERROR message CONTAINS "return code:500"
⚠️ 注意：WARN 告警的配置直接取决于你自己的日志采集系统的 logger 字段格式。 线上采用 %logger{12} 缩写，上述正则中的 c\\.f\\.f\\..* 字段可根据实际缩写形式调整。

🛠 技术方案与决策

决策说明
不引入 source 归因字段代码层无法可靠区分 EOF 根因（客户端断开可经 stream 透传到 Stone 调用被误判为 remote）。仅靠日志文案区分捕获位置。
CONNECTION_EOF 而非 CLIENT_UPLOAD_EOF按现象命名，不暗示根因。包含读请求体和远端 Stone 调用两类 EOF。
TurboFilter 旁路重打而非 DENY保留 WARN 日志可查（含原 logger name、message、堆栈、MDC），供外部告警。
统一异常层 EOF 返回 500 而非 400远端 Stone EOF 原本走 500，降级日志不改变接口返回码语义。
NO_FILE_PERMISSION_CODE 仅精准匹配 code 而非所有 FsiClientException避免扩大降级范围，不掩盖其它真实下游异常。

✅ 结论

当前治理效果
  • 已消除约 71,000+ 条 / 7 天的 ERROR 噪音，包括全部的 stream no originImageData（~15,000）、全部 EOF（~48,000）、uploadForKuaiXiao 两类（~1,388）、NO_FILE_PERMISSION_CODE（~86）。
  • 剩余 ERROR 约 2,700 / 7 天，其中值得告警的仅 Read timed out 和 Stone 500（~1,941），其余零散需逐条分析。
  • 所有降级仅影响日志级别，不影响业务：HTTP 返回码、异常传播路径、成功路径行为均未改动。
  • 预存失败测试与本次无关：ProcessCaptchaHandlerRouteTest、FsFscCgiDubboConfigTest、DocPreviewProxyTest 在干净树上同样失败。
  • 已治理的报错屏蔽后问题很小：均为可预期的客户端/业务输入失败或连接 EOF 常态噪音，单条不可动作，由外部日志采集对 WARN 设立趋势告警即可。
