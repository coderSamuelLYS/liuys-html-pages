迁移方案 · 企业粒度灰度

非网盘大文件存储
从阿里云 OSS 迁移到腾讯云 COS

历史对象由腾讯云 CDM 完成跨云搬运（copy 语义、源数据保留），本服务负责存储路由切换与对账。按企业分批灰度，切前补迁 + 切后补迁兜底所有窗口期。

1目标与边界

把非网盘、非数据专属的大文件默认存储从 OSS 迁到 COS。CDM 负责搬运，本服务只做路由切换与对账，不实现跨云搬运。

网盘业务不迁移

继续走 OSS（前端不支持切换）

数据专属 / 专属存储企业显式 s3Type

不参与默认迁移，由配置控制

上传不中断容忍短暂真空

切换瞬间落在旧存储的对象，靠切后补迁扫掉

本服务职责仅路由 + 对账

不实现跨云搬运，跨云交给 CDM

2整体策略

三步走：先全量迁大头，再按企业分批灰度，最后全局切默认存储。

迁移前提（均已确认）

🔑
objectKey 迁移前后不变

CDM 保持原 objectKey，不加额外目录；同 key 在 OSS 与 COS 指同一对象

📋
CDM 为 copy 语义

迁移后源 OSS 数据保留，对象两边共存

⚡
objectKey 一次性写入不覆盖

CDM"同名已存在则跳过"策略在此前提下幂等安全

🎯
objectKey 含企业信息

CDM 可按企业 objectKey 规则精确匹配本批迁移范围

💡

核心路由模型：因 objectKey 不变且对象两边共存，路由只需依据「企业当前存储类型」（灰度名单 / 全局默认），不逐对象判断，也不需要逐对象路由真值字段。正向切流、回退、多实例配置时间差产生的"落旧存储未 copy"对象，统一由切后补迁扫掉。

每批灰度节奏

→

3灰度控制

新增批量灰度名单，不复用 enterprise-toml 逐个写 section。

# 灰度名单配置
cos.gray.enterprise.list = EA001,EA002,EA003

需要改造点

  • EnterpriseS3TypeHelper 加载灰度名单
  • 配置热更新时刷新名单
  • /api/getEnterpriseS3Type 可用于验证企业是否命中 COS
  • 命中灰度名单时记录日志，便于排查

4执行步骤

① 准备 COS

  • 创建 COS Bucket；配置 fs-bfm-tencent-cos
  • 验证 COS 上传、下载、abort
  • 验证 /cos/callback/upload 能正常写入 file_meta
  • 确认 CDM 目标路径保持原 objectKey，不加额外目录

② 全量迁移

  • 创建全局 CDM 迁移任务，把 OSS 历史对象迁到 COS
  • 导出失败对象列表，处理完成后再进入灰度
  • 基础校验：对象数量、总大小、失败列表、抽样下载

说明：网盘或专属对象若被一起迁到 COS，不代表业务读 COS；只要存储选择逻辑正确，它们仍走原存储。

③ 批次灰度

选定本批企业
→
触发 CDM 迁移 + 对账（切前补迁）
→
校验 COS 存在、失败数为 0
→
加入灰度名单
→
再次触发 CDM（切后补迁）
→
验证 + 保留 ≥24h

批次建议

批次企业范围目的
第 1 批1-2 个内部/测试企业验证全链路
第 2 批5-10 个小流量真实企业验证生产稳定性
第 3 批30-50 个中等流量企业放大验证
第 4 批剩余企业持续追加直到全量

④ 全局切换

所有批次完成后，低峰期执行：

# 全局切换
defaultBigFileObjectStorage = cos
cos.gray.enterprise.list = ""

显式 s3Type 配置继续保留，用于网盘外的专属存储或个别企业回退。

5批次准入

切换前必须满足准入条件；切换后须满足晋级判据方可进入下一批。

切换前准入

  • 切前已触发本批企业 CDM 迁移任务，并完成本批对象校验
  • 本批企业失败对象为 0；若存在失败对象，含失败对象的整个企业移出本批（路由为企业级，无法逐对象排除）
  • 灰度名单变更后，已准备好立刻再次触发本批迁移任务
  • 本批企业没有 appendObject 流量（当前 COS 不支持 appendObject 授权）
  • COS Callback 验证通过；抽样下载验证通过

切换后晋级判据（任一未达 → 触发批次回退）

≥ 24h
观察时长（覆盖一个业务高峰 + 夜间周期）
≥ 99.9%
下载成功率（按对象数）
≥ 99.5%
Callback 成功率
≤ 0.1%

6回退

正向与回退机制对称：都依赖「两边共存 + 补迁兜底」。

🔵 批次回退

  • 从 cos.gray.enterprise.list 移除异常企业
  • 等配置热更新生效，企业回到 OSS
  • 触发反向 CDM 任务，把切换期间写入 COS、OSS 尚无的增量 copy 回 OSS（与正向切后补迁对称，不可省略）
  • 暂停后续批次，保留已成功批次观察

🔴 全局回退

  • 改回 defaultBigFileObjectStorage = oss
  • 启动反向迁移（COS → OSS），补切换期间写入 COS 的对象
  • OSS 老数据一直保留，反向只需补增量，量小
  • 停留越久 COS-only 增量越大 → 回退越早越好，需快速监控判据支撑早发现
  • 反向完成前，这部分新文件可能无法经 OSS 默认路径访问

7风险

  • 不停上传会有短暂真空期；每批采用「切前补迁 + 切后立刻补迁」降低影响
  • appendObject 当前只支持 OSS/self，灰度企业不能有 append 上传流量
  • OSS 不能整体只读，因为网盘仍需写 OSS
  • COS Callback 失败时需要人工补齐 file_meta 或重试 Callback

8最终稳态

非网盘、非数据专属COS
网盘OSS
数据专属 / 专属存储企业显式 s3Type
OSS保留网盘能力
⚠️

不可逆决策点：「两边共存 + 补迁兜底」仅在双副本共存期成立。降成本 = 删 OSS 副本，删除后全局回退将从「仅迁增量」退化为「70TB 全量回迁」。删除前必须确认已稳定观察 ≥2 周且无 COS 侧异常，建议分批删除：先删非网盘非专属历史副本；网盘副本随网盘迁移另行评估。
