核心 Blocker代码事实灰度瞬切

头号问题：下载路由不读 file_meta.bucket，企业进灰度名单即全量历史文件瞬时切到 COS

代码核实发现，普通文件下载时并不会根据文件实际所在 bucket 路由，而是根据当前企业灰度/配置解析出 s3Type，再直接签发当前云厂商 bucket 的下载凭据。因此一旦企业进入 cos.gray.enterprise.list，该企业历史文件下载会全部指向 COS。

这意味着：CDM 迁移失败、漏迁或切换前刚写入 OSS 尚未同步的对象，都会在灰度生效后出现下载 404；且当前没有运行时自动回退 OSS 的安全网。

代码事实链路

以下是本次 review 的关键证据。重点是 file_meta.bucket 主要被写入，但普通下载路径不消费它。

下载路由

S3ManagerImpl.getDownloadAuthInfo
→ resolveS3Type(ea, business, isNetdisk)
→ enterpriseConfig.getBucketName()

普通下载基于当前配置/灰度解析云厂商，不基于对象实际落点。

bucket 字段

FileManagerImpl / CallbackController / CosUploadFinalizeService
→ 仅 business 场景常见写入 file_meta.bucket

本次目标「非网盘非业务专属普通文件」大量历史记录 bucket 可能为空。

当前迁移接口

DataMigrationServiceImpl
→ OSSClient.copyObject
→ updateBucketByFilePath

这是阿里云 OSS 桶内 copy 逻辑，与 CDM 托管跨云搬运不是同一职责，应在新方案中明确作废/改造。

回退风险

CosClientFactoryImpl
→ cache expireAfterWrite(3500s)
→ changed() 未 invalidateAll

灰度名单删除后，部分实例可能继续签 COS 近 58 分钟。

关键问题清单

按上线阻断程度整理为 5 个 Blocker 与 8 个 Major。

下载无双读兜底

灰度名单生效后历史对象全量指向 COS。若 CDM 漏迁/失败/延迟，用户下载直接失败。建议 COS 404 自动回退 OSS 签发。

COS 授权失败被吞

上传/下载授权异常仅 warn，可能返回空凭据导致客户端静默失败。建议抛明确业务异常并加指标告警。

回滚缓存窗口过长

COS 客户端缓存 3500 秒，配置变更未主动失效；删除灰度名单后仍可能签 COS。建议 TTL 降 60 秒并在 changed() invalidateAll。

旧迁移接口与 CDM 方案不匹配

/migration/tranBucket 无鉴权、硬编码业务、使用 OSSClient.copyObject。建议标记废弃，或改造成对账/准入接口。

Callback 失败无补偿

回调写 file_meta 失败只依赖日志。建议幂等表、重试、定时补录，避免对象已上传但元数据缺失。

Major 问题概览

主题问题建议
CDM 对账CDM 完成后如何确认 COS 对象就绪、失败列表如何入库未定义。CDM 任务 ID、失败对象列表、head COS 校验、0 失败准入。
appendObject方案要求批次企业无 append 流量，但没有可执行预检。按 ea 聚合 7/30 天 append 记录，>0 不准入。
可观测性路由决策日志不足，线上无法快速判断为何走 COS/OSS。记录 ea/appName/business/isNetdisk/source/finalS3Type。
监控上传/下载指标未按 s3Type 分桶。加 s3Type tag；新增 COS_DOWNLOAD_404、CALLBACK_FAIL。
晋级回滚批次推进标准不量化。观察 ≥24h；错误率、404、callback fail 设置硬阈值。
反向迁移回退到 OSS 前，新 COS 文件可能在 OSS 不可访问。继续双读兜底，或给反向迁移对象数/吞吐/ETA。
网盘边界网盘识别只看 appName + 企业灰度，业务维度边界需确认。明确网盘/非网盘/业务专属路由矩阵。
备份计费withOutOurCosEnterprise 与 OSS 侧灰度 key 语义需核对。补备份、bizLog、计费验证用例。

推荐方案

不建议让 BFM 承担跨云搬运。保留 CDM 作为搬运方，但必须在 BFM 侧补「对账准入 + 双读降级 + 回滚缓存失效」。

方案 A · 推荐

COS 404 回退 OSS

下载主走 COS；若对象不存在或签发失败，自动用 OSS 配置二次签发。改动小，最适合作为灰度安全网。

方案 B

单对象路由字段

新增/使用 storedS3Type 或 bucket，迁完一批对象再逐对象切路由。更精确，但需要数据回填和更大改造。

必须配套

CDM 对账准入

CDM 任务完成后拉取失败对象列表，head COS 抽样/全量校验，失败为 0 才允许企业进灰度名单。

建议实施时间线

阶段 0：术语与职责对齐

统一迁移产品名（CDM/CMS/MSP），明确「云产品负责搬运，BFM 负责对账、路由、降级」。

阶段 1：发布安全网

实现 COS 下载 404/授权失败 → OSS 回退；授权失败抛异常；COS 客户端缓存变更主动失效。

阶段 2：CDM 全量预迁

为批次企业跑全量任务，导出失败对象；失败对象入库并重试，直至失败为 0。

阶段 3：切前增量补迁

切换前窗口跑增量补迁，确认近实时写入对象追平；appendObject 流量预检为 0。

阶段 4：灰度切流

加入 cos.gray.enterprise.list，观察下载 404、callback fail、上传/下载错误率、P99 延迟。

阶段 5：观察与晋级/回滚

观察不少于 24h。指标达标晋级下一批；异常时删除灰度名单并依赖缓存失效 + OSS 回退兜底。

最终结论

方案方向可行：使用腾讯云托管迁移产品做跨云搬运，按企业灰度切换，是合理路径。

但不能直接进入批次 1：当前代码中灰度名单是企业级全量历史文件的瞬时切换开关，且下载不读对象真实落点。必须先补 COS→OSS 双读兜底、CDM 对账准入、回滚缓存失效和授权失败告警。
