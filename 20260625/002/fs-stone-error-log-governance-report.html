四、EOF 治理详情（本轮重点落地）

背景：fileserver 的 ClientAbortException / stream2byte exception 基线 7 天 ~50,257 条（占该服务 45.7%），本质是客户端在服务端写流完成前断开（用户取消下载、网络中断），非服务端故障。

两层治理：AOP 兜底 + Service 层堵漏

层治理点文件机制来源

ProxyResourceProfileproxy/aop/ProxyResourceProfile.java（新建）前序会话

新增 isClientAbort() NFileServiceImpl.java 私有方法，复用 AOP 同款 EOF 判定（遍历 cause 链 10 层） 本次

AOP 切点覆盖范围

AOP 类切点表达式覆盖的接口类
FileResourceProfile execution(* ...fileserver.resource.*.*(..)) NFileResource / AbstractFileResource / AFileResource / DefaultFileResource
ProxyResourceProfile execution(* ...proxy.resource.*.*(..)) ProxyResource / NProxyResource / AProxyResource
FsiResourceProfile execution(* ...proxy.adapter.resource.*.*(..)) NFileStorageAdapter / EnterpriseStorageAdaptor / XTCompatibleAdapter
⚠️ 覆盖关系关键点：AOP 切的是 Resource 层，只对抛出到 Resource 层的异常有效。Service 层 catch 后吞掉的异常（makeWater 原来吞掉继续执行、uploadAvatar 原来 return）到不了 AOP，必须改业务代码堵漏——这正是本轮 Service 层 3 处改动的作用。

EOF 判定逻辑（isClientAbort）

识别 5 种客户端断连特征，遍历异常 cause 链最多 10 层：

  • ClientAbortException — 类名包含
  • java.io.EOFException — 类名等值
  • Unexpected EOF read on the socket — message 包含
  • Broken pipe — message 包含
  • Connection reset by peer — message 包含

五、WARN 日志统计口径

logback.xml 配置：root level=info → ASYNC → RollingFile，业务类无单独 logger 覆盖。log.warn(...) 正常落盘，logger 名 = 类全限定名（@Slf4j 默认）。

可直接按 logger 统计的 WARN（Service 层显式，本次新增）

logger行WARN 文案触发场景
com.facishare.stone.fileserver.
service.impl.NFileServiceImpl		 783 empty stream for watermark,request:{} makeWater 读到空流
900 uploadAvatar client aborted,request:{} uploadAvatar 客户端断连

建议统计查询

logger = com.facishare.stone.fileserver.service.impl.NFileServiceImpl
  AND level = WARN
  AND (msg LIKE 'empty stream for watermark%' OR msg LIKE 'uploadAvatar client aborted%')

AOP 层降级的 WARN（~5万条 ClientAbort）走 rpc-trace 框架上报，不在本地 RollingFile 里。建议用 log_error_dist 中 ClientAbort ERROR 量下降做反向印证。
